# Mossy Wave

A serverless notes board built entirely on the **AWS free tier**, demonstrating a production-grade architecture with infrastructure-as-code.

## Architecture

```
User
 └─▶ CloudFront (HTTPS, OAC, cache policies)
       ├─▶ S3  ──────────────────── React + Vite frontend
       └─▶ API Gateway (HTTP API)
             └─▶ Lambda (Node 20, inside VPC)
                   └─▶ DynamoDB (GSI, TTL, PAY_PER_REQUEST)
                         ▲
                   VPC Gateway Endpoint (free — no NAT Gateway)
```

### Free-tier services

| Service | Why |
|---|---|
| S3 | Static frontend hosting |
| CloudFront | CDN + HTTPS — Origin Access Control (OAC) |
| API Gateway (HTTP) | 1M calls/month free |
| Lambda (Node 20) | Business logic, runs inside VPC |
| DynamoDB | Notes table with GSI + TTL |
| VPC + Gateway Endpoint | Private subnets, no-cost DynamoDB access |
| EC2 t2.micro | Bastion host (optional, ≤ 750 hrs/month) |
| IAM | Least-privilege roles |
| ACM | Free TLS certificate for CloudFront |
| CloudFormation | Generated by CDK |

## Project layout

```
mossy-wave/
  cdk/          AWS CDK app (TypeScript) — 5 focused stacks
  api/          Lambda handlers (TypeScript)
  frontend/     React + Vite SPA
  .github/workflows/deploy.yml   CI/CD via GitHub Actions + OIDC
```

## CDK stacks

| Stack | Resources |
|---|---|
| `MossyWaveNetworking` | VPC, subnets, DynamoDB gateway endpoint, bastion |
| `MossyWaveStorage` | S3 bucket, DynamoDB table + GSI |
| `MossyWaveCompute` | Lambda functions, least-privilege IAM role |
| `MossyWaveApi` | API Gateway HTTP API, routes, CORS |
| `MossyWaveCdn` | CloudFront distribution, OAC, cache policies |

## Local development

### Prerequisites
- Node 20+
- AWS CLI configured (`aws configure`)
- CDK bootstrapped (`cdk bootstrap`)

### Build & deploy

```bash
# Build Lambda handlers
cd api && npm ci && npm run build

# Build frontend
cd frontend && npm ci && npm run build

# Build & deploy CDK
cd cdk && npm ci && npm run build && npx cdk deploy --all
```

### Local frontend dev

Set `VITE_API_URL` to your deployed API Gateway URL, then:

```bash
cd frontend && npm run dev
```

## CI/CD

GitHub Actions deploys on every push to `main` using **GitHub OIDC** — no long-lived AWS keys stored as secrets. Only `AWS_DEPLOY_ROLE_ARN` is required in repository secrets.

### One-time GitHub OIDC setup

1. Create an IAM OIDC provider for `token.actions.githubusercontent.com`
2. Create an IAM role with the trust policy scoped to your repo
3. Add `AWS_DEPLOY_ROLE_ARN` to GitHub repository secrets

## Security highlights

- S3 bucket is fully private — CloudFront OAC is the only access path
- Lambda runs in isolated private VPC subnets
- DynamoDB reached via free VPC Gateway Endpoint (no internet egress)
- IAM roles follow least-privilege (no wildcard actions)
- No long-lived AWS credentials in CI (OIDC only)
- Notes auto-expire after 30 days via DynamoDB TTL
